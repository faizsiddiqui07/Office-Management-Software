## 🔴 Critical & High (verified — fix first)

### 1. [HIGH · bug] dashboard
**File:** `website/app/(app)/dashboard/page.jsx`

The leadership 'Pending approvals' StatCard displays analytics.pendingApprovals.length (page.jsx line 254), but the backend caps that list at 6 (LeaveRequest.find(...).limit(6) in backend/src/services/dashboard.service.js line 75). With 15 pending requests the card reads '6', silently under-reporting. The team snapshot correctly uses countDocuments, so the two numbers can even contradict each other on the same screen for a manager+leadership user.

**Fix:** Have the backend return a real pendingApprovalsCount (it already runs countDocuments for team.pendingApprovals) and bind the StatCard to that, keeping the 6-item list only for the preview card.

### 2. [HIGH · bug] dashboard
**File:** `website/app/(app)/dashboard/page.jsx`

Chart header 'Monthly spend ({new Date().getFullYear()})' (page.jsx line 271) hardcodes the calendar year, but the backend builds monthlyExpenseTrend from `${year}-01-01..${year}-12-31` where year = leaveYearOf(todayYMD) — the FISCAL leave year label (backend/src/services/dashboard.service.js lines 21, 74). From Jan 1 to Mar 31 the dashboard shows LAST year's expense data under the current year's title. Wrong data labeling for 3 months of every year.

**Fix:** Decide one convention (calendar year is natural for expenses), fix the backend range to use the actual calendar year, and/or have the backend echo the range so the frontend title is derived from data, not new Date().

### 3. [CRITICAL · consistency] attendance
**File:** `website/components/attendance/regularization-queue.jsx`

12-hour AM/PM formatting via a local to12() helper (lines 23-28), used to render requested check-in/out times in both PendingList (line 71) and HistoryList (line 155): `In ${to12(r.requestedCheckIn)}`. Directly violates the owner's 24-hour-everywhere rule — approvers see '6:00 PM' here but '18:00' everywhere else in the app.

**Fix:** Delete to12(). requestedCheckIn/requestedCheckOut are already 24h 'HH:mm' strings — render them as-is, or add a shared formatHM() to lib/time.js if zero-padding normalization is wanted.

### 4. [CRITICAL · consistency] attendance
**File:** `website/components/attendance/my-regularizations.jsx`

Second duplicated to12() helper (lines 12-17), used on line 36 to show the employee's own correction request times in 12h AM/PM. Same 24h-rule violation, plus the helper is copy-pasted rather than shared.

**Fix:** Remove to12() and print the raw 24h 'HH:mm' value (or the shared lib/time helper). One shared formatter kills both violations.

### 5. [HIGH · mobile] attendance
**File:** `website/components/attendance/regularization-queue.jsx`

Approve / Reject / Delete buttons in the corrections queue all use size="sm", which is h-7 (28px) per components/ui/button.jsx — well under the 40px touch-target floor. These are the queue's primary actions, tapped on Android phones, and Approve sits 8px from Reject.

**Fix:** Use default size (or className="h-10 sm:h-8") for Approve/Reject/Delete rows, and keep more horizontal gap between Reject and Approve (gap-2 → gap-3) on mobile.

### 6. [HIGH · ux] leaves
**File:** `website/components/leaves/requests-queue.jsx`

Approvers can never see the employee's leave reason. The RequestsQueue columns are requester/type/range/days/status/actions — there is no 'reason' column — and the Approve/Reject decision dialog description only shows `${name} · ${days} day(s) · ${type}` (no reason, no dates). An approver must approve or reject blind, or ping the employee on WhatsApp to ask why.

**Fix:** Add the reason to the decision dialog body (and the date range: `formatRange(...)`), and add a reason column (line-clamped) or make rows clickable to open a detail view, matching the attendance module's row-click detail modal pattern.

### 7. [HIGH · ux] leaves
**File:** `website/components/leaves/leave-history.jsx`

Employees never see the approver's decision note or who decided. RequestsQueue collects an optional note on approve/reject ('Add a note for the employee…'), but LeaveHistory columns are type/range/days/reason/status/actions only — a REJECTED row gives zero explanation, so the collected note goes nowhere visible to the employee.

**Fix:** Show the decision note and decider on the history row (extra line under status, or row-click detail dialog like attendance). If the backend doesn't return it on GET /leaves, that needs adding too.

### 8. [HIGH · mobile] todo-tasks
**File:** `website/components/tasks/task-board.jsx`

The primary interaction — the click-to-complete circle in TaskRow — is 'size-5' (20px) with only 'mt-0.5' offset and no padding/hit-slop. On a 375px Android phone this is half the 40px minimum touch target, and it sits directly next to tappable row content, so employees will mis-tap when ticking off work.

**Fix:** Keep the 20px visual circle but wrap it in a larger hit area, e.g. add 'p-2 -m-2' or make the button 'size-10' with the ring/check drawn on an inner size-5 span.

### 9. [HIGH · feature] todo-tasks
**File:** `website/components/tasks/task-board.jsx`

Tasks cannot be edited after creation. TaskDialog fully supports edit mode (task prop, PATCH /tasks/:id, 'Edit task' title) but grep shows it is only ever rendered as '<TaskDialog />' with no task prop (task-board.jsx line 311), and TaskRow renders only a delete button — no edit affordance. An employee who typos a title or needs to move a due date must delete and re-create; an assigner cannot fix a delegated task at all (assignee can't delete it either, per line 107 comment).

**Fix:** Add an edit action on TaskRow (pencil icon or tap-row) that opens the already-built TaskDialog in edit mode; extend the same to the assigner's folder view for delegated tasks.

### 10. [HIGH · ux] people
**File:** `website/components/users/users-directory.jsx`

"Reset credentials" fires instantly from the row dropdown with no confirmation (resetMut.mutate(row.original.id) directly in DropdownMenuItem onClick, lines 130-134). One mis-tap — very likely on a phone dropdown — immediately invalidates the employee's current password and locks them out until the new temp password is shared. Deactivate gets a ConfirmDialog; reset does not.

**Fix:** Reuse the existing ConfirmDialog pattern (like the toggling state) — 'Reset credentials for {name}? Their current password will stop working immediately.' with tone=destructive.

### 11. [HIGH · consistency] people
**File:** `website/app/(app)/team/page.jsx`

The Team page component has no permission gate at all, but the nav item is gated: lib/permissions.js line 66 declares { label: 'Team', href: '/team', permission: 'viewEveryone' } (managers+ only). Any employee who types /team directly sees every person's email and phone. Contrast with users/page.jsx which correctly gates with can(user, 'createUsers') and renders a 'No access' EmptyState.

**Fix:** Add the same guard as users/page.jsx: const allowed = !!user && can(user, 'viewEveryone'); render an EmptyState if not allowed (and confirm the backend GET /users enforces it too — the frontend gate alone is cosmetic).

### 12. [HIGH · bug] visitors
**File:** `website/components/visitors/visitor-table.jsx`

Search fires a server request on every keystroke with no debounce, and because each keystroke changes the queryKey ({ queryKey: ['visitors','list',filters] }) with no placeholderData/keepPreviousData, isLoading flips true and the whole table is replaced by TableSkeleton on every character typed. On an Android PWA over mobile data this means request spam plus a flashing table while searching.

**Fix:** Debounce the search value (~300ms) before it enters `filters`, and pass `placeholderData: (prev) => prev` (TanStack v5 keepPreviousData) so the previous rows stay visible while refetching; show a subtle isFetching indicator instead of the skeleton.

### 13. [HIGH · ux] visitors
**File:** `website/components/visitors/visitor-table.jsx`

The `purpose` field is captured in VisitorDialog ('Purpose / notes' Textarea) but is never displayed anywhere — no table column, no row-click detail. The only way to read why a visitor came is to open the Edit dialog. There is also no onRowClick detail modal, unlike the attendance module which recently added one.

**Fix:** Add onRowClick opening a read-only detail sheet/modal showing all fields incl. purpose (mirroring the attendance row-detail pattern), or at minimum a truncated purpose column with line-clamp.

### 14. [HIGH · feature] visitors
**File:** `website/components/visitors/visitor-table.jsx`

No one-tap 'check out now' action. Real front-desk flow: visitor leaves, receptionist must tap Edit, open the dialog, focus the time input and type the time. Rows with an empty checkOutTime just render '—' with no affordance, and there is no indication of who is currently still in the office.

**Fix:** For rows where checkOutTime is empty, render a small 'Check out' button (fills nowHM() via a PATCH) and a 'In office' tone badge instead of '—'.

### 15. [HIGH · consistency] money
**File:** `website/lib/expense.js`

Expense summaries and charts use the CALENDAR year, not the Indian fiscal year. lib/expense.js currentYearRange() returns `{ from: `${y}-01-01`, to: `${y}-12-31` }`, which drives the 'This year' StatCard (expense-summary.jsx), both charts ('Monthly · {year}' in expense-charts.jsx) and the top-category stat. The owner's standing convention is Apr–Mar fiscal year (already enforced for leave), and an Indian firm's expense book is reconciled Apr–Mar. From Jan–Mar the 'This year' total silently mixes two financial years.

**Fix:** Add fiscalYearRange() (Apr 1 – Mar 31, matching the leave-year helper) and use it in ExpenseSummary and ExpenseCharts; label the chart 'FY 2026–27'.

### 16. [HIGH · mobile] calendar-announce-activity
**File:** `website/components/calendar/holiday-calendar.jsx`

Month-grid calendar is near-unusable on a 375px phone: 'grid grid-cols-7 gap-1' with 'min-h-[84px]' cells yields ~44px-wide cells; event chips are 'text-[10px]' with 'truncate' and 'px-1.5 py-0.5' (~21px tall touch target), so titles render as 3-4 unreadable truncated characters and '+N more' is plain text, not tappable.

**Fix:** On tap of a day cell, open a bottom-sheet listing that day's events in full (title, range, type, description) for ALL users; keep chips as color dots only on <sm screens. Optionally default mode to 'agenda' on mobile via a matchMedia check.

### 17. [HIGH · ux] calendar-announce-activity
**File:** `website/components/calendar/holiday-calendar.jsx`

Non-admins cannot view event details at all in month view: day-cell onClick is 'isAdmin ? setDialog(...) : undefined' and chip onClick only fires 'if (isAdmin)' (lines 119, 136). For a regular employee, tapping a truncated chip does nothing — the description field is only reachable via the Agenda toggle, which nothing hints at.

**Fix:** Make chips/day cells open a read-only detail popover or bottom-sheet for everyone; show the edit dialog only when isAdmin.

### 18. [HIGH · bug] calendar-announce-activity
**File:** `website/components/calendar/upcoming-holidays.jsx`

UpcomingHolidays fetches only the current calendar year ('/holidays?year=${year}') then filters endYMD >= today. In late December (or any time near year end) January holidays of the next year never appear and the card claims 'No upcoming holidays this year' — bad for an Apr–Mar fiscal-year firm where Q4 (Jan–Mar) always crosses the calendar-year boundary of a December viewer.

**Fix:** Query a rolling window instead: '/holidays?from=<today>&to=<today+90d>' (the endpoint already supports from/to — birthday-popup.jsx uses it), or fetch year and year+1.

### 19. [HIGH · ux] settings-roles-profile
**File:** `website/app/(app)/profile/page.jsx`

Profile photo can only be set via a raw 'Avatar URL' text input (Input id="p-avatar", placeholder "https://…"). Employees on Android phones have no way to host an image and paste a URL — effectively nobody gets an avatar. The codebase already has lib/image.js downscaleImage() and a working upload pattern (settings logo upload posts a dataUrl), so this is an inconsistency as much as a gap.

**Fix:** Replace the Avatar URL input with a tap-to-upload flow: hidden <input type="file" accept="image/*">, run downscaleImage(file, { maxDim: 256 }), POST the dataUrl to /auth/profile, mirroring uploadLogo() in settings/page.jsx lines 123-145.

### 20. [HIGH · ux] settings-roles-profile
**File:** `website/app/(app)/settings/page.jsx`

Settings page mixes two save models with no dirty-state protection: one global 'Save settings' button at the very bottom (line 650-654) covers Organisation, Work hours, Check-in alerts, GPS and Expense categories, while Email (SMTP) has its own 'Save email settings' button mid-page. On a phone this page is 6 panels of scrolling; a user who edits work hours and then taps 'Save email settings' (or navigates away) silently loses those edits — there is no unsaved-changes indicator, no sticky save bar, no beforeunload/route guard.

**Fix:** Track dirty state (compare form to data.settings) and render a sticky bottom save bar (fixed bottom-0 with safe-area padding) that appears only when dirty, plus a toast/confirm when navigating away with unsaved changes. Visually separate the SMTP panel as 'saved independently'.

### 21. [HIGH · ux] shell-auth-foundation
**File:** `website/app/(auth)/forgot-password/page.jsx`

Forgot-password success screen shows developer-only copy to end users in production: "No SMTP configured in dev? The link is printed to the backend console." (line 53). Employees on their phones will read an internal debugging note.

**Fix:** Remove the hint entirely, or gate it behind process.env.NODE_ENV !== 'production'.

### 22. [HIGH · bug] shell-auth-foundation
**File:** `website/app/(auth)/login/page.jsx`

The login page's "Remember me" checkbox (lines 176-179) is a decoy: it has no state, no name, and is never sent to the API. Users who untick it expecting a non-persistent session get identical behavior.

**Fix:** Either wire it to session persistence (short-lived vs long-lived token/cookie via the login payload) or delete the checkbox.
